## Why

对象存储改造上线后，线上出现：公开博客列表像「没有文章」、已发布详情打不开、写作台「保存草稿」点击无反馈。根因集中在存储写失败未暴露、列表页把接口错误当成空列表、以及正文对象缺失时读路径过脆。需要尽快修复读写与错误可见性，恢复可用的发布/浏览闭环。

## What Changes

- 修复公开博客列表：接口失败时展示明确错误，不再伪装成「没有找到文章」
- 修复保存草稿 / 发布：对象存储不可用或校验失败时，写作台给出可读错误与成功反馈（避免「点了没反应」）
- 加固正文读取：`content_key` 文件缺失时回退遗留 `content_md`（若有）；仍无正文时返回可理解业务错误，避免详情白屏/假 404 混乱
- 创建/更新写对象前校验存储根可写；失败信息映射为前端可展示的文案（如存储不可用）
- 补充运维自检：确认 `BLOG_STORAGE_ROOT`、目录权限、迁移是否完成；必要时提供一键/文档化的内容修复步骤
- **不做**：回退整套对象存储方案；改公开 URL；引入云 OSS

## Capabilities

### New Capabilities
- `blog-read-write-reliability`: 博客公开读与作者写在对象存储模式下的错误可见性、存储失败处理、正文加载回退与运维自检

### Modified Capabilities
- `blog`: 公开列表在加载失败时 SHALL 展示错误态，不得仅显示空列表
- `blog-persistence`: 正文加载与写入在对象缺失/存储不可用时的行为与错误码约定（与作者保存反馈对齐）

## Impact

- **前端**：`BlogListView` / 写作台错误与成功提示；可选首页 latest 同样处理
- **后端**：`BlogPostService.loadContent` 回退策略；存储错误文案；必要时启动/健康检查提示
- **运维**：`BLOG_STORAGE_ROOT` 权限与迁移；根 README 可补一行故障条目
- **依赖**：沿用现有 `BlogObjectStore`，无新中间件
